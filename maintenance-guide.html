一、每日操作：每次更新博客的标准流程

部署之后，你的博客不是一锤子买卖——你会写新文章、改老内容、调样式、修链接。每一次改动，都遵循一个固定流程。这个流程你做过一次就记住了，本质上就是三件事：改本地 → build → push。

标准更新流程（每次改动必走的三步）

改内容
在本地用任何编辑器改你的文件——可能是改 Markdown 文章、改 CSS、改 HTML、替换图片。
如果你改了 posts/*.md（文章），改完后必须跑 python build.py（下面会讲为什么）。
如果你只改了 js/articles.js（文章配置），也要跑 build。
如果你只改了 HTML 或 CSS 文件——不需要跑 build，直接跳到第二步。
构建（如果需要）
打开终端，在博客目录下运行： 
python build.py
如果改了历史文章（history-posts/），也要跑： 
python build_history.py
这两个脚本的作用是把 Markdown 文章"打包"进 JS 文件，让网页能直接读。不跑的话，网页里还是旧内容。
推送上线
终端里跑三行命令： 
git add .
git commit -m "更新了第18篇文章的标题"
git push
commit 信息写清楚你改了什么——不要写"update"或者"修改"，写具体：
✅ "第18篇删除'但是是真的吗'"
✅ "工具页图标换成 SVG"
❌ "fix"
❌ "update"
写清楚的好处：三个月后回来看，你能一眼知道这次提交干了什么。别人接手也能看懂。

推完之后：

GitHub Pages 会在 1-5 分钟内自动部署更新。打开你的网站地址，按 Ctrl+Shift+R（强制刷新，跳过浏览器缓存），确认改动生效。如果没变，再多等 2 分钟——GitHub 的部署管道偶尔会排队。

什么时候不需要跑 build？

只有改这些文件时才需要：posts/*.md（文章内容）、history-posts/*.md（历史幕）、js/articles.js（文章配置）、pages/*.md。因为 build.py 的任务就是把这些 Markdown 转成 JS。

以下改动不需要 build：

  • 改 HTML 文件（index.html、agent.html 等）
  • 改 CSS 文件（style.css、portal.css 等）
  • 改 JS 文件里的逻辑（app.js、history-app.js 等，但 articles.js 例外）
  • 替换图片（SVG、PNG 等）
  • 改 README.md

改了 JS/CSS 之后别忘了版本号

你的博客用 ?v=YYYYMMDD 来控制缓存。如果你改了 js/app.js 或 css/style.css，找到引用该文件的 HTML 页面（通常是 agent.html、index.html、history.html），把版本号从 ?v=20260703 改成当天日期 ?v=20260704。这样读者浏览器才会加载新版本，而不是用缓存里的旧文件。

⚠️ 忘了递增版本号 = 读者看到的还是旧样式/旧逻辑。浏览器会以为文件没变，继续用缓存。这是最常见的问题——"我改了 CSS 怎么页面没变？"——答案 90% 是忘了改版本号。

二、Git 基础：你只需要会的 5 个命令

你不用学 Git 的分支、合并、变基。一个单人博客，master 就是你唯一的战场。你需要的命令就这 5 个：

命令作用什么时候用
git status 看哪些文件被改了、哪些还没提交 每次改完东西、在 add 之前先跑一下，确认改动的文件清单是你预期的。如果看到意外文件（比如自动生成的临时文件），考虑加 .gitignore。
git add . 把所有改动标记为"准备提交" 确认 git status 无误后跑。如果想只提交某个文件：git add 文件名
git commit -m "信息" 创建一个提交（保存点） add 完了就 commit。信息写清楚你干了什么。
git push 把本地的提交推送到 GitHub（触发上线） commit 完了就 push。第一次部署之后，后续只需要 git push，不需要再加 -u 参数。
git log --oneline -5 看最近 5 条提交记录 想回顾你做了哪些改动时用。

如果写错了 commit 信息怎么办？

在 push 之前可以改：git commit --amend -m "新的正确信息"。如果已经 push 了——别管它，一条错误的 commit 信息不影响任何功能，下次注意就好。

如果 git add 加了不该加的文件？

用 git reset HEAD 文件名 把它从暂存区移除。然后如果这个文件以后也不想提交，加到 .gitignore 里。

三、常见更新场景

场景 1：新增一篇 AI Agent 文章 最常见

在 posts/ 下新建 .md 文件
命名规范：22-文章标题slug.md。不要写 # 标题——标题由 articles.js 渲染。
在 js/articles.js 里加配置
在 ARTICLES 数组末尾加一个新对象，包含 slug、title、tags、file 等字段。
跑 build
python build.py
递增版本号
在 agent.html 和 index.html 里，把 articles.js 和 content.js 的 ?v= 改成当天日期。
推上线
git add . && git commit -m "新增第22篇：XXX" && git push

场景 2：修改一篇已有文章的内容 高频

直接编辑 posts/XX-xxx.md → 跑 python build.py → git add . && git commit -m "修改第X篇：改了XXX" && git push。如果没有改动 articles.js 配置，不需要递增 articles.js 版本号，但要递增 content.js 版本号。

场景 3：修改 CSS 样式

改对应的 CSS 文件 → 在引用了该 CSS 的 HTML 文件里递增版本号 → git add . && git commit && git push。不需要跑 build。

场景 4：替换或新增 SVG 图片

把新的 .svg 放到 images/ → 在对应的 HTML 或 Markdown 里引用 → git add . && git commit && git push。不需要跑 build，除非 svg 被 Markdown 文章引用（Markdown 文章的图片路径会被 build.py 处理）。

场景 5：新增一个工具分类页

复制 tools-hardware.html 作为模板 → 改标题、内容、卡片数据 → 在 tools.html 的蜂巢区加一个新六边形链接 → git add . && git commit && git push。不需要跑 build。

四、build 脚本：什么时候跑、跑错了怎么办

build.py 做了什么

它读 posts/ 下所有的 .md 文件，读 pages/ 下的 about.md，把内容拼成一个巨大的 JavaScript 对象，写到 js/content.js。你的 agent.html 页面加载 content.js，就能直接从 JS 变量里取文章内容渲染——不需要在浏览器里发 HTTP 请求去读 .md 文件。这是你的博客在 file:// 下也能跑的原因。

build_history.py 做了什么

同样的逻辑，处理 history-posts/ 下的七幕文章，输出到 js/history-content.js。

跑 build 之前确认一件事

确保你的终端当前目录是 blog/（即包含 build.py 的目录）。不是 Workspace 根目录、不是上级目录。可以先用 ls build.py 验证——如果报文件不存在，说明目录不对。

build 报错了怎么办

常见报错 1：FileNotFoundError——某篇文章的 .md 文件找不到。

原因：js/articles.js 里某篇文章的 file 字段指向的文件不存在（可能改了文件名但没更新配置）。

修：去 js/articles.js 里找到对应的 article 对象，检查 file 字段路径是否正确。

常见报错 2：UnicodeEncodeError——文章里有特殊字符。

原因：某篇 .md 文件里有 build.py 不认识的编码字符。

修：检查最近改动的文章，确保另存为 UTF-8 编码。用 VS Code 打开文件，右下角状态栏会显示编码，如果不是 UTF-8 就点它改成 UTF-8。

五、缓存问题：改了东西但网页没变？

这是你未来最常遇到的"伪问题"——你以为部署成功了，其实是浏览器缓存了旧版本。按顺序排查：

先用 Ctrl+Shift+R 强制刷新
这个快捷键告诉浏览器"忽略缓存，全部重新加载"。80% 的情况这一步就解决了。
检查 GitHub Actions 部署状态
打开你的仓库 → Actions 标签页。看看最近的 workflow 是绿色✓还是红色✗。如果是红的，点进去看日志——大概率是 build 步骤失败。
检查版本号有没有递增
如果你改了 CSS 或 JS 文件，确认 HTML 里的 ?v= 确实改成了最新日期。如果还是旧日期，浏览器会继续用缓存。
确认 git push 确实成功了
打开 GitHub 仓库网页，看最新 commit 时间戳是不是你刚才 push 的时间。
等够时间
GitHub Pages 部署需要 1-5 分钟。如果刚 push 完立刻刷新没看到变化——再等 2 分钟。

一个检查部署是否成功的小技巧：

在网站 URL 后面加 ?random=12345（随便一串数字），比如 https://astoria77.github.io/AIdreamer/?x=1。这跟 Ctrl+Shift+R 一个原理——告诉浏览器"这是新 URL，别用缓存"。

六、排错速查表

症状可能原因怎么修
页面白屏 JS 报错导致渲染中断 按 F12 打开浏览器控制台，看 Console 里的红色报错信息。常见：content.js 格式错误、articles.js 语法错误（少了一个逗号或括号）。
文章列表没有新文章 忘了跑 build.py 或没递增版本号 跑 build → 递增版本号 → push → 等 2 分钟 → Ctrl+Shift+R
CSS 样式全丢了 CSS 文件路径错误或 CDN 挂了 F12 → Network 标签 → 刷新，看有没有红色的 404。确认 HTML 里的 <link href="css/style.css"> 路径对。
图片裂了 图片路径错误 你的博客用相对路径（images/xxx.svg），确保文件确实在 images/ 目录下且文件名完全一致（大小写敏感）。
标签页 / 搜索页功能异常 改了 app.js 逻辑但没测试 F12 控制台看报错。很多时候是修改了标签数据但忘了同步改 app.js 里的 tagClass() 或 graphNodeColor()。
git push 报 403 认证过期 运行 cmdkey /delete:LegacyGeneric:target=git:https://github.com 清除旧凭据，下次 push 重新授权。
git push 报冲突 远程仓库有本地没有的提交（极罕见） 单人博客不会遇到。如果遇到了：git pull --rebase 再 git push。

七、日常运维检查清单

每次改完内容后（即查即改）

  • ✅ 跑了 build.py（如果改了文章）
  • ✅ 递增了相关 CSS/JS 的版本号
  • ✅ git status 确认改动的文件都是你预期的
  • ✅ git push 成功（没报错）
  • ✅ 浏览器 Ctrl+Shift+R 看到改动生效

每周（5 分钟）

  • 打开你的博客网站，随便点几个页面，确认都正常
  • F12 控制台看一眼有没有红色报错
  • 检查 GitHub 仓库的 Issues 或 Actions 页面有没有异常

每月（10 分钟）

  • 备份：你的代码在 GitHub 上本身就是备份。但可以把整个 blog/ 文件夹压缩一份存到网盘或移动硬盘——纯粹的心理安慰，不花时间。
  • 检查 CDN 依赖：你的博客用了 marked.js 和 highlight.js 的 CDN。偶尔去 cdnjs.com 搜一下这些库，看看你锁定的版本 4.3.0 和 11.9.0 是否已经被标记为 deprecated（废弃）。如果被废弃了，换一个推荐版本。
  • 检查 Google Fonts 加载速度：在国内网络下打开你的博客，看标题字体是否有明显的「先显示系统字体再切换到衬线字体」的闪烁。如果加载超过 3 秒，考虑换国内字体 CDN 或回退到系统字体。

每季度（15 分钟）

  • 清理 dead code：你有没有在 posts/ 或 images/ 下放了不再使用的文件？如果有，删掉再 commit。保持仓库干净。
  • 检查链接：你的文章里有没有外链（指向其他网站的 URL）？如果有，点几个看看是否还活着。外链失效是最常见的「内容老化」。
  • 手机端测试：用手机打开博客，确认排版没乱。你改 CSS 的时候可能只在大屏幕上测过。

八、未来你可能想做的事

绑定自己的域名 中期

目前你的网站地址是 astoria77.github.io/AIdreamer。如果想换成 aidreamer.me 这种自己的域名：买域名（Namesilo / Cloudflare 约 ¥50-80/年）→ 在域名服务商那里添加一条 CNAME 记录指向 astoria77.github.io → 在 GitHub Pages Settings 的 Custom domain 填你的域名 → 勾选 Enforce HTTPS。整个过程 20 分钟，但域名购买需要海外支付方式。

加访问统计 中期

如果你想知道有多少人在看你的博客、哪篇文章最受欢迎：注册一个 Google Analytics 账号 → 在你博客的每个页面 <head> 里加一段 GA 提供的 JS 代码 → push 上线。或者用更轻量的替代品如 Plausible（隐私友好，但付费）或 Umami（开源，需自部署）。注意：加统计之前想清楚——你是想了解读者还是想被数字绑架？符合你"不为流量焦虑"的理念。

迁移到 Vercel 远期

如果将来 GitHub Pages 在国内访问太慢，或者你想用 Vercel 的自动构建（让平台帮你跑 build.py，不用本地跑），随时可以切。你的 vercel.json 已经准备好了——在 Vercel 里导入同一个 GitHub 仓库，1 分钟内完成切换。你的域名 astoria77.github.io 不冲突——GitHub Pages 和 Vercel 可以同时跑，你只是改一下 DNS 指向就行。

加 RSS 订阅 简单

如果有人想通过 RSS 阅读器订阅你的博客更新：你需要生成一个 feed.xml 文件，里面包含最新文章列表。可以用 Python 脚本自动从 articles.js 生成。这个不到 50 行代码就能搞定。

优化 Google Fonts 国内加载 简单

目前你的博客字体从 fonts.googleapis.com 加载，国内某些网络慢。你可以换到 fonts.googleapis.cn（Google 官方中国镜像），或者把字体文件下载到本地放在 fonts/ 目录下，用 @font-face 直接引用本地文件——彻底脱离对 Google 的依赖。

九、总结：一个简单的心理模型

运维你的博客，就是把三件事做对

1. 改对。知道自己改了哪类文件（Markdown/HTML/CSS/JS），知道是否需要跑 build、是否需要递增版本号。

2. 推对。git add → commit → push，三步一个节奏。commit 信息写清楚。

3. 验对。push 完打开网页，Ctrl+Shift+R 确认改动生效。如果没变，等 2 分钟再试。

不需要学的东西：Git 分支管理、GitHub Actions 配置、CI/CD 流水线、Docker、Kubernetes、Nginx 配置。这些是你三年后如果管 10 个项目的团队才需要考虑的事。现在，一个人、一个博客、一套静态文件——上面的三件事就够用了。

你的博客是一个活的东西。它不是部署完了就放在那里积灰。你每写一篇新文章、每修改一个颜色、每修一个错别字——都是一次小型运维。把这份手册放在手边，出问题了不慌，按索引查就行。
