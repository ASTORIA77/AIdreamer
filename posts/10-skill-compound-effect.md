Skill 这个东西，我最直观的感受就是一个字：**省**。省时间、省 token、省精力。你把一件事的流程打包好，以后每次都按同一个范式去执行——这就是复利效应。一次投入，后面无限次零成本复用。

这种感觉实在是太好了。所以我给这篇文章取了这个标题。

## 一、Skill 是人和 Agent 之间的协议

为什么说 Skill 是「协议层」的一部分？先理清三个协议各自的方向：

- **MCP** 是 Agent 和工具之间的协议——让模型能接上外部数据、API、文件系统
- **A2A** 是 Agent 和 Agent 之间的协议——让不同的智能体能互相协作
- **Skill** 是人和 Agent 之间的协议——你就告诉它：这件事该怎么做、按什么步骤、哪些红线别碰

Skill 本质上就是你和 Agent 之间的一份协议。你在告诉它：

- 面对某类任务时，你该用什么步骤去做
- 你可以用哪些工具
- 哪些红线你不要碰

它不是一次性指令，是沉淀下来可以复用的 SOP。它要解决的核心矛盾是：**普通大模型像一个极聪明的新员工，但它不知道你们公司的标准操作流程。给她一本《作业手册》，她就能按标准干活。**

## 二、渐进式披露：装了 100 个 Skill，启动也只占几百 token

Skill 最精妙的设计在机制层面——**Progressive Disclosure（渐进式披露）**。

你的 Agent 可能装了 100 个 Skill——有处理财报的、有写代码的、有翻译论文的。但每次启动时，模型**只载入每个 Skill 的 name 和 description**——就几十个词。等它判断当前任务匹配了某个 Skill，才把完整的 SKILL.md 正文读进来。真正开始执行了，才去调附件里的脚本和参考资料。

三个阶段，像查字典：

1. **发现**——只看目录（name + description），省 token
2. **加载**——翻到那一页（读正文），针对性获取指令
3. **取用**——需要时才查附录（跑脚本、读参考文件）

100 个 Skill，启动成本只有几百个 token。这就是渐进式披露的价值——你只为你真正用到的内容付费。

## 三、怎么创建一个 Skill

现在有四种主流方式：

**第一种：自己手写。** 最原始也最灵活。建个文件夹，写一个 `SKILL.md`——前半段是 YAML 元数据（name、description），后半段是 Markdown 正文（角色、工作流、约束、引用文件路径）。需要的话再加 `scripts/` 放脚本、`references/` 放参考资料。description 是整份文件最重要的字段——它决定了模型什么时候会唤起这个 Skill。

<img src="images/skill-directory.svg" style="max-width:100%;display:block;margin:16px auto;" alt="Skill 目录结构"/>

**第二种：用 Skill Creator。** 什么叫「用技能创建技能」？你告诉 AI 你想做一个什么样的 Skill，功能是什么、步骤是什么、避什么坑，它自己帮你把 SKILL.md 写好、目录建好。你负责审核，它负责组装。

**第三种：从社区装现成的。** Skill Hub 和 GitHub 上已经有大量别人写好的 Skill，直接下载放到你的 Agent 里就能用。生态起来了，你不用每个东西都从零写。

**第四种：你用的桌面 Agent 内置的。** 很多 Agent 软件本身就带了一套预装技能包——先去翻一翻它自带的东西，有用的直接启用。

还有一个越来越流行的路径：有些智能体会在你完成一个任务之后，主动问你——"你是不是想把这个流程变成一个 Skill？" 国内的话 Qoder Work 可以，国外 Hermes 也可以。这是一种非常自然的进化——你先用对话把流程跑通，跑顺了再封装成 Skill。

## 四、为什么我喜欢复利效应

回到标题。

Skill 最打动我的不是技术上的优雅——是它把「一次性劳动」变成了「可以滚雪球的资产」。你今天花了半小时写好了一个写周报的 Skill，以后每周写周报的时候都省了那半小时。你今天写好了一个翻译论文的 Skill，以后每篇论文的翻译质量和格式都保持一致。

打包一次，复用永远。而且复用的是你自己沉淀下来的最佳实践，不是别人写的通用模板。

## 五、还有一个隐藏价值：逼你拆解自己的业务流程

Skill 写得好不好，衡量标准只有一个——**你对自己工作流程的理解有多深**。

这其实是 Skill 带来的一个隐藏训练：你得把自己的工作拆解成一步一步——哪些是输入、哪些是决策点、哪些是重复动作、哪些容易出错的坑——然后把这些变成一份清晰的指令集。每写几步就行，短而精准的几个步骤远比一份冗长手册更好维护。

我写 Skill 的时候，一般按同一个框架来拆：**CODE 原则**——Capture（收集）、Organize（组织）、Distill（提炼）、Express（表达）。它本质上就是知识管理中的 DIKW 金字塔的一次实践：原始数据 → 有结构的信息 → 可复用的知识 → 能指导行动的智慧。你会发现这四个环节全程都在做同一件事——把信息往上层提。每次提炼，信息的密度和价值都在增加。

举个例子：我要写一个"整理会议纪要"的 Skill，按 CODE 拆就是——先 Capture：录音转文字、把散乱的原始对话全捞进来；再 Organize：按议题归类、筛掉闲聊和重复的话；然后 Distill：提取每个议题的结论、待办、风险点；最后 Express：输出一段结构化的纪要和一条艾特各负责人的消息。

你会发现，大部分日常工作提炼到知识层（能复用、能指导行动）就已经是非常满意的交付物了。能到智慧层（抽象出通用原则）是可遇不可求的，不强求。

所以 Skill 不只是在教 AI 怎么干活——它也在教你把自己的思路一层层理清楚。理清楚了，写出来的 Skill 才短、才准、才真正能跑起来。
在这个过程中你就被迫做一件事：**模块化**。一个大任务拆成几个小 Skill，每个 Skill 只管一件事。学会了拆解，你才能真的享受到复利——因为小模块更容易被复用、更容易被组合、出错时也更容易定位问题。

所以我感觉，Skill 不只是在教 AI 怎么干活——它也在教你怎么把自己的思路理清楚。

在 AI 的世界里，比「能陪你聊天」更高级的词只有一个——**省心**。Skill 就是通往省心的最短路径。
