> 服务器与客户端之间的通信协议——会用但讲不好的东西，今天查资料把它弄清楚

## 一、先坦白

我不是计算机专业的。

API 这个东西，我用是天天在用。WorkBuddy 调用大模型走的是 API，各种软件之间的数据交换走的也是 API，甚至我在浏览器里打开一个网页，背后也跑着一堆 API 调用。但我要是被问到「所以 API 到底是什么？」——我会卡住。

我知道它是个通信协议。我知道它的作用就是让两个系统之间搞清楚「谁跟谁、要干什么」。但再多说几句就开始露怯了。

这篇文章就是我老老实实去查资料、理解、然后用自己的话讲一遍的过程。如果你也跟我一样——不是科班出身，但对 API 有个模糊的感觉又说不清楚——这篇应该适合你。

## 二、API 是什么——一个服务员的比喻

我找到的最好的比喻，是把 API 想象成餐厅里的**服务员**。

你去餐厅吃饭。你不能直接冲进厨房自己炒菜——厨房有厨房的规矩，你也不知道食材在哪。你得通过服务员：你告诉服务员你要什么，服务员把你的需求翻译成厨房能懂的语言，厨房做好了菜，服务员再端给你。

在这个比喻里：

- **你** = 客户端（你的浏览器、你的 App、你的 WorkBuddy）
- **服务员** = API
- **厨房** = 服务器（真正干活的地方，比如 OpenAI 的 GPU 集群）
- **菜单** = API 文档（告诉你什么能点、怎么点）
- **点菜** = 请求（Request）
- **上菜** = 响应（Response）

所以 API 本质上是一个**约定好的通信方式**——它定义了「你按什么格式说话，服务器就按什么格式回应」。不是黑魔法，就是一套规矩。

## 三、API Key：一把钥匙开一把锁

那问题来了：服务员怎么知道你是正经顾客，不是来蹭空调的？

这就到了 **API Key**。它就像一把钥匙——你有这把钥匙，才能打开对应服务的门。

具体来说：当你在 OpenAI 或者 DeepSeek 申请 API 使用权限时，平台会给你一串长字符串，像这样：

```
sk-abc123def456ghi789jkl...
```

创建 API Key 本身超级简单——登录大模型 API 平台（DeepSeek、豆包、阿里云百炼等），找到 API Key 管理页面，点一下 `➕` 创建就行。生成的 Key 基本都是 `sk-` 开头的。

**但有个关键动作：创建完立刻复制保存。** 很多平台出于安全考虑，API Key 只在创建时展示一次，关掉页面就再也看不到了。如果没及时存，只能删了重建。建议存到密码管理器或者本地的加密文本里，别随手扔微信聊天记录里。

然后你每次发请求的时候，都要把这串钥匙带在请求里（通常是放在一个叫 `Authorization` 的 Header 里，写成 `Bearer sk-xxx...` 的格式）。服务器看到这把钥匙，就做三件事：

1. **确认你是谁**（认证）
2. **确认你有没有权限**（鉴权）
3. **记录你用了多少**（计费）

没有钥匙？门都进不去。钥匙过期了？同样的结果。

这个设计的好处很明显：API 是公开的端点（谁都可以往那个地址发请求），但只有带了有效钥匙的人才能拿到结果。不靠钥匙的话，服务器根本不知道要不要理你——总不能随便来个人就免费给算力吧？

## 四、实际操作：Headers、Base URL、POST、GET

说了半天比喻，来看看实际长什么样。假设你要让 ChatGPT 帮你写一段代码，你发出去的请求大概是这样的：

```
POST https://api.openai.com/v1/chat/completions
Headers:
  Content-Type: application/json
  Authorization: Bearer sk-your-api-key-here

Body:
{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "user", "content": "用 Python 写一个冒泡排序"}
  ]
}
```

这里面几个关键零件：

**Base URL** = `https://api.openai.com/v1`——这是服务器的地址，相当于「服务员在哪」。每个 API 服务商有自己的地址：
- OpenAI：`https://api.openai.com/v1`
- DeepSeek：`https://api.deepseek.com/v1`
- 你本地的 Ollama：`http://localhost:11434/v1`

**Header** = 请求的「附加说明」。最常用的两个：
- `Content-Type`：告诉服务器「我发的是 JSON 格式的数据」
- `Authorization`：带上 API Key，告诉服务器「我是谁」

**Body** = 请求的「具体内容」。这里放了你要问什么、用什么模型、以及各种配置参数。

**请求方法**（GET vs POST）：
- **GET**：问服务器要数据。比如「把今天的天气给我」——参数通常拼在 URL 里：`?city=beijing`
- **POST**：给服务器发数据。比如「帮我生成一段文字」——数据放在 Body 里

调用大模型用的是 POST。GET 只适合简单的查询。

用命令行工具 `curl` 的话，整个请求就一行：

```bash
curl https://api.deepseek.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-xxx" \
  -d '{
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "你好"}]
  }'
```

就这，没别的了。服务器收到这个请求，跑一遍模型，返回一个 JSON 格式的回复——里面就是大模型生成的文字。

不过真实世界里，你通常不用自己写这么一段。如果你用的是一个支持自定义模型的智能体框架，它会把 Headers、Content、Role 这些底层细节封装起来，最后只给你留一个表单：

<img src="images/api-config-form.svg" alt="API 配置表单：框架把底层细节打包好" width="100%"/>

这就是我觉得 API 最亲切的地方：**复杂的东西在下面，你看到的界面只有几个输入框。**

## 五、API 最让我心动的一点：便宜

现在再说说钱的事。

我一直觉得 WorkBuddy 的订阅费挺合理的——58 块钱一个月，等于买了一个会帮你干活的高级助手。但就在最近，它涨价了：标准版到了 99 元/月。这个消息让我重新算了一笔账——**直接调用 API 的价格，便宜到离谱**。

查了一下 2026 年最新的大模型 API 定价。国内模型的价格已经卷到了离谱的地步：

| 模型 | 输入价格（每百万 token） | 输出价格（每百万 token） |
|------|-------------------------|-------------------------|
| DeepSeek V4 Flash | 约 ¥1.0 | 约 ¥2.0 |
| 小米 MiMo-V2.5 | 约 ¥1.0 | 约 ¥2.0 |
| DeepSeek V4 Pro | 约 ¥3.1 | 约 ¥6.3 |
| 小米 MiMo-V2.5-Pro | 约 ¥3.1 | 约 ¥6.3 |
| MiniMax M3 | 约 ¥2.2 | 约 ¥8.6 |

**日常使用我肯定首选 DeepSeek V4 Flash**——输入一块钱，输出两块钱，国内访问稳定，中文理解在线，还能自己配置进智能体框架。自掏腰包学 AI，它就是性价比之王。

但话说清楚：我不是不碰国外模型。遇到真正需要最强能力的场景——复杂推理、写关键代码、做深度分析——我会毫不犹豫切到 Claude Opus 或 GPT-5.5，哪怕它们贵几十倍。**我一向觉得该花钱的地方就要用最好的**，只要这笔钱用对地方就行。

便宜的守住底线，贵的撑住天花板，这才是理性的折腾。

还有一个省钱细节值得说说：**现在流行高峰涨、低峰降的动态定价**。比如 DeepSeek 在白天高峰期价格会往上浮一点，深夜凌晨低谷时段则更便宜。阿里 Qwen 3.7 Max 也是类似的策略——低峰期直接打折，相当于鼓励你在不那么挤的时候用最好模型、按低价算。

所以如果你不着急要结果，深夜跑批量任务能省不少。这也算 API 比订阅制灵活的一个维度——你不仅可以选择用什么模型，还可以选择什么时候用。

一百万 token 是什么概念？一本《三体》三部曲加起来大概 90 万字、120 万 token。也就是说，用 DeepSeek API 读完整套《三体》并生成总结，可能花不到三块钱。

那是不是应该全面转向 API，不再用订阅制 Agent？

我的判断是：**这取决于你要买的是什么。**

WorkBuddy 的 99 元/月，不只是模型调用的钱——它还包括了 Harness 工程化框架（工具调用、记忆、编排、约束）、网盘存储、自动任务等功能。我在[第 5 篇](index.html#/post/what-you-pay-for)详细拆过。如果你需要的是一个开箱即用的工作系统，每天重度使用，不想自己接工具、写脚本、搭工作流——那这个订阅是划算的。

API 则是另一回事：你买的是**裸模型调用**。它便宜，但需要你自己把工具、记忆、任务调度这些环节串起来。对于使用量不大、或者只是偶尔需要深度推理的人来说，直接走 API 确实更划算；但如果你每天都靠它处理复杂任务，省下来的模型调用费可能还不够你搭这些基础设施的时间成本。

说白了：**订阅制买的是省心，API 买的是自由。**

而且 API 还有一个隐藏好处：**几乎没有门槛**。不像订阅制有月费，API 是充多少钱用多少。你今天注册一个 DeepSeek 账号，充 10 块钱，可能够用一个月的轻量对话。10 块钱在现在的物价下是什么概念？一杯奶茶都买不到。

这让我想起一个数据：2026 年上半年中国新增的 AI 原生应用超过 120 万个，其中 70% 是基于低价 API 构建的。API 价格战把 AI 能力变成了水电一样的基础设施——开个水龙头就有，用量计费，不设最低消费。

对于我这样自掏腰包学 AI 的人来说，API 是真正的「降门槛」。

## 六、买之前一定看清楚：我的 180 元教训

API 便宜是便宜，但买套餐这件事有个坑，我亲自踩过。

大模型 API 的付费方式通常有两种：**按量计费**（充多少钱用多少）和**套餐包**（coding plan / token plan，一次性买断一定额度）。套餐包通常更划算——比如某平台 180 元买个 coding plan，平均下来单次调用比按量便宜一半。

但买之前我只看价格，没看一件事：**这个套餐的 API，能不能配到我用的智能体框架里？**

买完才发现，它的接口格式跟 OpenAI 不兼容，生态做得又不好，我用的框架根本不支持接这个模型。180 元就这么没了。钱不算多，但堵心是真的。

所以现在我就一个原则：**买 API 套餐前，先确认你用的智能体框架支持它。** 现在主流大模型 API（DeepSeek、豆包、OpenAI 兼容接口）基本都没问题，但有些小平台生态做得差，买了就是孤岛。下单前先在框架里把 Base URL、Key、模型名填进去，发一条测试消息能跑通再买。

多花 5 分钟，比买完之后接不进去强一万倍。

## 七、更灵活：Temperature、Top-K、Top-P

说起调整模型参数，我用的 WorkBuddy 其实做不到——它是封闭的订阅制桌面 Agent，你输入 Prompt、它返回结果，中间过程是个黑盒。但**有些接入 API 的软件可以做到**，这也是直接用 API 比封闭 Agent 更灵活的地方。

如果你用 API 调模型，就可以调整这三个参数：

**Temperature（温度）**——控制模型的「创造力」。
- `0.1`：像个考试学霸，永远只选最确定的答案。写代码、翻译、事实问答时用。
- `0.7`：像个专业创意写手，在合理范围内有点变化。通用对话时用。
- `1.2`：像个喝醉的诗人，天马行空，放飞自我。头脑风暴时用。

原理很简单：大模型每次生成下一个词时，它心里有一个「候选词概率表」。比如接下来可能是「北京」（85%）、「上海」（8%）、「深圳」（4%）……Temperature 的作用就是**把这个概率分布拉平或拉尖**。温度越低，高概率词的差距越大，模型越保守；温度越高，低概率词也被拉上来，模型越爱冒险。

**Top-K**——限定候选人数量。
比如设定 `Top-K=50`，模型只从概率最高的 50 个词里选，后面的统统忽略。简单粗暴，但有时候会把合理的冷门词也排除掉。

**Top-P**——动态设门槛。
比如设定 `Top-P=0.9`，模型从概率最高的词开始累加，直到总和超过 90%，然后只在这个词库里选。更智能，能自动适应不同场景。

大多数任务组合用 `temperature=0.3, top_p=0.7` 就够了；创作用 `temperature=0.8, top_p=0.95`。别被参数吓到，其实大部分时间默认值就挺好。

能调参数这件事的意义在于：**用订阅制 Agent 时你是乘客，用 API 时你是司机**。你可以精细控制模型的行为——这个自由度在某些场景下很重要。

## 八、本地部署 vs 云端：Base URL 一改就行

还有一个让我觉得 API 设计很妙的地方：**本地部署的大模型和云端大模型，用的是同一套接口**。

什么意思呢？比如有人在自家电脑上用 vLLM 或者 Ollama 跑了一个开源模型，它在电脑上开了一个服务，地址是：

```
http://127.0.0.1:8000/v1
```

然后你只需要把代码里的 Base URL 从 `https://api.openai.com/v1` 改成 `http://127.0.0.1:8000/v1`，API Key 随便填一个（本地不需要真正验证），就完事了。同样的代码，同样的格式，同样的请求体——**无缝切换**。

这个体验我虽然自己还没来得及试，但看了一下教程，流程简单到离谱：

```bash
# 1. 安装 Ollama
# 2. 拉模型
ollama pull qwen2:7b
# 3. 服务自动启动在 http://localhost:11434/v1
# 4. 在代码里把 base_url 一改，API key 填 "ollama"
# 5. 跑起来了
```

云端和本地的核心区别：

| | 云端 API | 本地 API |
|---|---------|---------|
| Base URL | `https://api.openai.com/v1` | `http://127.0.0.1:8000/v1` |
| API Key | 真实付费密钥 | 随便填（`ollama`/`vllm-local`） |
| 模型 | 厂商提供（GPT-4o 等） | 自己下载的开源模型 |
| 费用 | 按 token 计费 | 只有电费 |
| 隐私 | 数据上传到云端 | 数据不出本机 |

这个发现让我意识到：**OpenAI 无意中定义了一个行业标准**。现在几乎所有大模型服务——不管是云端的 DeepSeek、智谱 GLM、通义千问，还是本地的 vLLM、Ollama、LocalAI——都支持 OpenAI 兼容接口。你只需要把 URL 和 Key 换一下，剩下的代码完全不用动。

## 九、收个尾

这篇文章我写得很坦诚——因为我确实不是科班出身，不会用那些高级的架构图来说服你。我只能告诉你我作为一个每天用 API、但讲不清楚 API 的人，在查完资料之后的真实理解：

API 就是一套约定的通信格式——用一种标准化的方式，让不同的程序之间能互相聊天。API Key 是钥匙，Header 是附加说明，Base URL 是地址，Body 是你想说的话。

它最厉害的地方不是技术上的优雅——是经济上的民主。曾经只有大公司才用得起的 AI 能力，现在一个月五十块左右就够日常使用了。而 OpenAI 无意中建立的接口标准，让云端的 ChatGPT 和本地的开源模型能用同一套代码调用——这个东西本身，就值得好好珍惜。

写到这我已经很满足了。把之前模模糊糊懂的东西查资料理清楚，本身就是一个学习过程。
