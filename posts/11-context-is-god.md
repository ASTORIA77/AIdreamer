> 现阶段 Agent 好用不好用，关键不在模型多强——在你多会给它搭语义背景。

## 一、什么是 Context——不只是一句 prompt

"Context"这个英文词翻译过来是「上下文」或「语境」，但我觉得最准确的叫法是**语义背景**。它不只是你敲进去的那句话（prompt），而是模型在开口之前看到的**全部信息**。

看到没有？prompt 只是冰山露在水面上的那个小尖儿。水面之下，才是真正决定 Agent 输出质量的东西。

有个很形象的类比——Andrej Karpathy 说过：**大模型相当于操作系统的 CPU，上下文窗口相当于 RAM。** CPU 强不强当然重要，但 RAM 里装了什么、装得乱不乱、关键信息有没有，可能比 CPU 本身更重要。Context Engineering（上下文工程）就是管理这块 RAM 的活儿——什么时候存、什么时候取、什么时候清理、什么时候压缩。

但 RAM 有一个硬限制：**满了就满了。** 对话越来越长，上下文窗口迟早会被撑满。这时候 Agent 有三种处理方式：

**自动压缩。** WorkBuddy、Qoder Work 这些主流 Agent 都能自己感知到上下文快满了、主动触发压缩。以WorkBuddy 为例，它有两层——第一层是轻量级的，每发一条消息前检查一次，超过一定比例就把最近对话整理成精简版；第二层是深度压缩，上下文持续涨到 70%-92% 阈值时触发，把旧消息做结构化总结、变成摘要嵌进上下文。我自己的体验是，任务中调用了一堆工具、返回了大量数据的时候，哪怕聊得不久也会触发——这就是 context rot：不是对话本身长，是工具输出的数据塞满了窗口。压缩不是魔术——是让模型把长内容概括成短的。中间总会丢信息，只是丢的是你不需要的。

**手动压缩。** Qoder Work 今年 6 月（0.6.3 版本）新增了手动压缩按钮——输入框右侧有个用量指示器，点一下就把历史对话压成摘要。不需要等系统自动触发，你觉得自己当前的对话够长了就随时压。适合长项目的中间节点——一个阶段做完了，压缩一次，再继续下一阶段。

**直接清掉。** 如果你的上下文联系不大——上一个任务是写周报，下一个是查天气——直接 Clear / New Chat。新任务开新会话是零成本的性能优化，不需要让 Agent 带着你在一个很长的上下文窗口里跑无关任务。

## 二、我是怎么给它搭语义背景的

做为一个以学习为主的知识工作者，我目前用的就是这四招：

**第一招：接上我的知识库** 我在 IMA 里构建了一个个人知识库，然后把它打通给 Agent。当 Agent 知道我库里有什么的时候，它检索出来的东西就不再是大模型凭空生成的，而是我亲手筛选过的信息。举个例子——我现在在学 AI，如果告诉 Agent「我喜欢吴恩达的课」或者「我在看周志华的西瓜书」，它给我写学习资料的时候就不再生成通用的教程，而是知道按谁的框架来、用谁的例子、讲到哪个深度。

学习场景是这样，编程场景也一样——写代码的人最爱把 GitHub 仓库接成 Context，项目结构、commit 历史、issue 记录全是语义背景，Agent 看一眼就知道技术栈、代码风格、当前卡在哪，不用每次从头解释。

说到这儿必须提一句飞书。飞书在 Agent 时代特别火不是没道理的。2026 年 3 月，飞书官方开源了 **lark-cli**——把 2500+ API 全部变成命令，内置 26 个 Agent Skills（日历、消息、文档、多维表格、邮箱、任务、会议全覆盖），MIT 协议，一行 npm install 就能用。背后的逻辑很简单：**你的飞书，就是你的工作语义背景。** 你要参加的会、群里的消息记录、多维表格、文档、审批、任务列表——全在飞书里。Agent 直接读懂这些，不需要你复制粘贴——它就已经知道你在忙什么、跟谁协作、下一步要干什么。OpenClaw 社区最早拥抱飞书，接上之后 Agent 真能帮你干活：搜文档、读妙记、查日历、发消息、调多维表格数据。**Agent 和你的工作环境连通的那一刻，你不再需要给它搭 Context——你的工作环境本身就是 Context。**

**第二招：直接传附件** 这个最直接——把参考资料丢给它。一个 PDF、一份课程大纲、一篇论文。模型不会去猜你要什么，它拿到实物放在眼前。

**第三招：把工作空间定在我自己的目录** Agent 默认会开一个自己的工作空间，但你如果手动把工作空间指向你自己的某个文件夹，那里面所有的文件它就都能读了。项目代码、实验记录、之前的笔记——不需要每次都上传，告诉它路径就行。

**第四招：README 交接**

一个项目做久了，对话越来越长，哪怕 compact 压缩之后也不太行了。这时候就让 Agent 出一份这个项目的 README——把项目目标、已完成的部分、当前卡在哪、下一步要做什么，清清爽爽写清楚。然后开一个新的对话（或换一个 Agent），把这份 README 扔进去，它就能直接接手。

这招有一个很奇妙的现象——是朋友告诉我的：Agent 写的 README，他这个人类拿着读不太懂，但是扔给另一个 Agent，对方能读懂。没有人类的惯性解读和预设，机器对机器的文本传输反而更干净。一份干净的项目 README，等于给新 Agent 配了一个零起点的上下文。

这个技巧的正经名字叫 **Agent Handoff（智能体交接）**。GitHub 上已经有专门做这件事的工具（agent-handoff-kit），核心逻辑就是让 Agent 生成一个结构化的 HANDOFF.md，包含启动命令、环境变量、目录结构、已知问题、待办事项。下一个 Agent 看一眼就上手，不需要再翻前面的几百轮对话。

## 三、Context 就是给模型的"作业要求"

其实这个东西用一个场景就能说清楚。

你去找老师交作业，说"老师帮我写一份报告"。老师不知道你要写什么、写多长、给谁看、用什么格式——出来一定是个泛泛的东西。

但如果你说："老师，我要写一份关于 Transformer 架构的学习报告，给刚学完 CNN 的同学看，控制在 2000 字左右，用论文《Attention Is All You Need》做主要参考，语气轻松一点，不要太多公式。"——老师接到这个作业要求，写出来的东西就是另一个档次了。

**Context 就是给模型的"作业要求"。** 你给得越清楚、越丰富，它做得越靠谱。这不是什么高级技巧，但做到了和不做的区别，就是「能用」和「好用」的区别。

## 四、为什么说 Context 是现阶段的神

提示词工程的热度在退，Context 的热度在涨。这不是炒作——是模型能力进到这个阶段之后，逻辑的自然走向。

2023 年，你得花大力气教模型怎么跟你说话。2026 年，模型本身已经够聪明了。瓶颈不在模型，而在**模型知不知道你到底要什么**。

Context 好的 Agent，像跟你共事了三年的同事——知道你早上习惯先看什么，知道你写报告的风格，知道你项目的来龙去脉。Context 差的 Agent，像一个博士毕业第一天来上班的新人——聪明，但是什么都不知道。

而现阶段，你自己就是那个决定 Agent 知道多少的人。你给它搭的语义背景有多厚实，它就有多好用。

这不是在讨论一个高级技术。这就是一个基本操作——每次开始干活之前，先问自己一句：**它现在的上下文里，有足够的信息来理解我吗？**
