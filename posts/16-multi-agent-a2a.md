> A2A 是 Agent 和 Agent 之间的协议，MCP 是 Agent 和工具之间的协议——各管各的，互不冲突。但说实话：多智能体听着很酷，我自己用起来却不是那回事。

## 一、多智能体为什么突然火了

2026 年，多智能体系统（Multi-Agent System）成了一个高频词。不管你打开哪个 Agent 的产品更新日志，几乎都能看到"团队协作""Agent Swarm""专家团"之类的字眼。

背后的逻辑其实很简单：**单个 Agent 的能力有上限。** 一个 Agent 同时搞定写代码、做设计、写文案、查数据、管项目——它确实能做，但精力分散了，质量会下降。所以行业共识是：让不同的 Agent 各管一摊，再让它们协作。

这就像公司里你不可能招一个"万能员工"，你得有设计师、程序员、运营、PM——各司其职。多智能体就是把这种分工模式搬进了 AI 世界。

现在主流桌面 Agent 里都有多智能体的影子：

- **WorkBuddy 的专家团**：可以组建多个专家（每个专家有自己的 Skill 和配置），复杂任务由不同专家协作完成。
- **Qoder Work 的团队协作**：支持多 Agent 并行处理，项目经理 Agent 分配任务，工程师 Agent 执行。
- **Kimi Work 的 Agent Swarm**：智能体集群模式，多个 Agent 自主协商、各取所长。

这些功能的出现，标志着 Agent 正在从"一个人干所有事"向"一群人分工合作"进化。

## 二、多智能体的三种组织方式

多智能体系统怎么组织？学术界和工业界基本分成三种架构模式：

<img src="images/multi-agent-architectures.svg" alt="多智能体的三种组织方式" style="max-width:100%;display:block;margin:16px auto;"/>

**中心化（主从式）**——一个"主"Agent 负责分配任务、调度资源、汇总结果，其他 Agent 只管执行。好处是简单可控，主 Agent 看全局；坏处是主 Agent 是单点瓶颈，它挂了全停。WorkBuddy 专家团、Qoder Work 项目经理模式，本质上都是中心化的。

**去中心化（对等式）**——没有"领导"，所有 Agent 平等，通过协商决定谁做什么。好处是没有单点故障，扩展性强；坏处是协调成本高，容易"吵架"。Kimi Work 的 Agent Swarm 就是这个方向。

**混合式**——上面有个"主"定大方向，但具体执行时允许 Agent 之间自行协商。结合了两种模式的优点，也是目前被认为最实用的方案。

不管你用哪种模式，核心问题都是同一个：**Agent 之间怎么对话？** 这就是 A2A 协议要解决的事。

## 三、A2A 协议：Google 给多智能体立的规矩

2025 年 4 月，Google 发布了 A2A（Agent-to-Agent）协议——第一个专为 AI 智能体间通信设计的开放标准。2026 年 3 月正式发布了 v1.0 版本。

A2A 解决的核心问题是：**不同厂商、不同框架、不同组织开发的 Agent，怎么互相发现、互相通信、互相协作？**

如果说 MCP（我在[第 12 篇](agent.html#/post/mcp-and-connectors)里写过的 Model Context Protocol）解决的是"Agent 怎么连上数据库、文件系统、API"——那么 A2A 解决的是"Agent A 怎么跟 Agent B 说：嘿，帮我查个东西"。

两者的关系是互补的：

| | A2A | MCP |
|---|---|---|
| **连的是谁** | Agent ↔ Agent | Agent ↔ 工具 |
| **交互方式** | 多轮协商、状态管理 | 单次调用、请求响应 |
| **典型场景** | 委托任务、分工协作 | 查数据库、调 API、读写文件 |

A2A 的核心设计有五个支柱：

1. **统一传输格式**：基于 JSON-RPC 2.0 over HTTP(S)。跟 MCP 一样，底层是标准协议，不是黑魔法。
2. **Agent Card 发现**：每个 Agent 对外发布一张"名片"——一个 JSON 元数据文档，写明自己的身份、端点、能力、技能列表。访问 `https://域名/.well-known/agent.json` 就能拿到这张名片。
3. **任务状态管理**：A2A 中的任务是"有状态"的。一个任务可以走完 `已提交 → 处理中 → 完成` 的完整生命周期，中间可能来回多次（比如 Agent 说"我还需要你补充一个信息"）。
4. **多模态支持**：消息里可以带文本、文件、结构化数据。不是只能传文字。
5. **企业级安全**：OAuth 2.0 认证、JWT 授权、TLS 加密——生产环境能用，不是玩具。

任务的状态流转长这样：

```
已提交（submitted）→ 处理中（working）→ 已完成（completed）
                  ↕                   → 失败（failed）
            需要输入（input-required）
```

当 Agent A 把任务丢给 Agent B，A 可以持续追踪 B 的状态——是在跑、是卡住了、还是做完了。这比简单的"发个请求等个回复"要复杂，但也更接近真实的工作协作方式。

如果 MCP 是给 Agent 配工具——锤子、螺丝刀、电钻——那 A2A 就是给 Agent 配同事。你有你的专长，我有我的专长，我们互相委托、互相配合。

## 四、但说实话：我没怎么用过多智能体

写到这儿我要坦诚一件很重要的事。

讲了半天多智能体、A2A、Agent Swarm——**但我在实际使用中，几乎从来不组建多智能体团队。**

不是我觉得它没用。是我觉得**太累了。**

你想，我现在用的这个 Agent，它已经被我调教得很好了——它知道我喜欢"干湿分离"，知道我不追前沿论文，知道我的项目结构，知道我的写作风格。它的 Memory 里存了我的偏好，它的 Skill 和 Context 是我花了很长时间一点一点堆出来的。

然后你让我再新建一个 Agent？重新告诉它我的背景、我的偏好、我的风格？给它装 Skill？写 User Prompt？配 MCP 连接器？

我每次想到这个步骤，就感觉像公司里来了一个新员工——得从"我们是做什么的"开始培训。然后我意识到：**这就是为什么领导总爱用一个人——不是偏心，是懒得再教一遍。**

我现在开始理解这个心态了。一个已经磨合好的 Agent，就像用顺手了的工具——你知道它哪里强、哪里弱、怎么说话它最听得进去。你换一个新的，所有这些默契都要重建。

所以我现在基本就是一个 Agent 打天下。把它的 Skill 装满，把 Context 搭厚，让它通过内部的子 Agent 调用（Sub-agent）来处理复杂任务——而不是去建一个"多智能体团队"。至少在我现在这个阶段，够用，也不累。

## 五、还有一个隐藏问题：信息耗散

多智能体系统还有一个工程上的老问题——**信息在 Agent 之间传递时，一定会失真和耗散。**

这是信息论的基本规律：每经过一次转述，信息就丢一点。你跟一个人说"帮我做三件事"，它记住了三件。但如果你跟一个 Agent 说"帮我协调另外三个 Agent 做这三件事"——每一个转述环节都可能在丢细节、变形、或丢失优先级排序。

不管你是中心化（主 Agent 往下传递指令），还是去中心化（Agent 之间来回协商），还是流水线式（Agent A 的产出喂给 Agent B）——信息只要过了一遍手，就会被压缩、被简化、被误解。

这不是工程实现的问题，是信息传递本身的天花板。

所以有时候，**一个知道所有上下文的人（或 Agent），可能比一个"各管一摊"的团队更高效。** 因为它不需要转述。不需要同步。不需要"你再补充一下这个信息"。

我现在的判断很简单：多智能体是趋势，但不是必须。当你面对的任务复杂度超过了单个 Agent 的认知上限，多智能体就是必需的——就像你一个人管理不了的部门得拆成团队。但在那之前，先把手上的一个 Agent 用好、用透、用出默契——可能比追着建一个"多智能体团队"更实在。
