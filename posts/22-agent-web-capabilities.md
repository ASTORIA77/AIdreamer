> 大多数人口中的「联网搜索」根本不是一回事——Agent 上网有三层，大部分人只用了最浅的一层

这篇其实本来不在计划内——21 篇已经把从入门到反思的闭环走完了。但前两天写了个课件，把 Agent 怎么上网页这件事系统性捋了一遍。写完觉得这个知识点太少有人讲清楚过，而且我刚好有个实操经验能派上用场，就再挤一篇出来。

## 一、先把概念搞对：上网 ≠ 搜索

当你说「让 AI 帮我搜一下 B 站上 Transformer 教程，按弹幕最多的排前三」——它能做到吗？

大多数人觉得可以。毕竟现在哪个 AI 不标榜「联网搜索」？

但这个东西本质上是三个完全不同的操作。能做到哪一步，取决于你给了 AI 哪种「眼睛」：

- **第一种：WebSearch（搜索 API）**——不打开任何网页，只问搜索引擎一句「你知道哪些结果？」，拿到一份标题 + 摘要的列表就回来交差。
- **第二种：WebFetch（页面抓取）**——把网页的 HTML 源码拿回来，转成文字，用一个小模型读一遍，告诉你「这个页面上写了什么」。
- **第三种：WebBridge（浏览器控制）**——直接操控你真实在用的浏览器，带着你的登录态，把页面完整渲染出来，甚至能帮你点按钮、填表单、截张图回来。

三种都叫「上网」，但看到的是三个不同的世界。

## 二、三个层次，用大白话讲清楚

### 第一层：WebSearch —— 相当于问图书管理员

你去图书馆问管理员：「有关于 Transformer 的书吗？」

管理员翻了翻目录卡片，告诉你：「有三本，书名分别叫《Attention Is All You Need 解读》、《Transformer 实战》、《图解 Transformer》，都在二楼 AI 区。」

你知道了书的存在，但你没翻开任何一本，不知道内容到底好不好、公式对不对、代码能不能跑。WebSearch 就是你雇的这个管理员——它访问的是搜索引擎的索引数据库，不是真实的网页。所以遇到 B 站搜索这种需要 JavaScript 动态加载的页面，它直接交白卷——搜索引擎的爬虫从来没索引过这些动态内容。

**WebSearch 回答的问题：**「有没有这方面的东西？」

### 第二层：WebFetch —— 相当于把书借回来自己读

你根据管理员给的线索，自己去二楼把三本书抱回来，摊在桌上从头读。书里的章节、公式、代码示例，全都看到了。但这里有一个前提——你抱回来的书必须是纸质印刷的。如果这书是一本只能在校内网在线看、需要学号登录的电子教材，你就抱不回来。

WebFetch 就是这样。它发一个 HTTP 请求（跟你在浏览器输入网址按回车是一回事），拿到 HTML 源码，转成文字，再用一个轻量模型读一遍、提炼重点给你。但 JavaScript 动态渲染的内容它看不到——因为 WebFetch 不执行 JS。现代的网页大部分内容都不是写在 HTML 里，而是前端 JS 从后端数据库动态拉出来、实时渲染到页面上的。WebFetch 拿到的，是渲染之前的「空架子」。

**WebFetch 回答的问题：**「这页面上写了什么？」

### 第三层：WebBridge —— 相当于你亲自去现场

不管那本书在不在图书馆、要不要登录、有没有反爬，你本人带着读者证走过去，刷卡进门，拿到书，看到了一切。

WebBridge 就是让 AI 借用你的浏览器。它不是一个新的浏览器实例——它就是你正在用的那一个。你的 Cookie（登录态）、你的插件、你的历史记录，全都在。AI Agent 通过一个本地守护进程发指令，浏览器扩展接到指令后在你的真实浏览器里执行——打开网页、滚动、点击、截图、甚至用 JavaScript 直接提取页面上的结构化数据。

同一个 B 站搜索页面 URL，三种工具看到的是三种完全不同的东西：

- **WebSearch 看到的：** 搜索引擎索引里一条 CSDN 博客的摘要
- **WebFetch 看到的：** 一个验证码页面（不带 Cookie，B 站直接挡在门外）
- **WebBridge 看到的：** 完整的视频列表，每个视频的标题、弹幕数、播放量、封面截图

三个工具，同一个地址，看到的不是一个东西。选错了工具，等于让近视的人不戴眼镜就去认路——他可能也走，但方向是错的。

## 三、浏览器获取内容的权限正在不断扩大

如果回头看一下 2025-2026 年这个赛道的演变，有一条线是很清楚的：AI 对浏览器的控制权限一直在升级。

最早的时候——大概 2025 年初——主流做法是「启动一个全新浏览器」。Playwright MCP、agent-browser、browser-use，都是这个路子。给你一个没登录过任何网站的空浏览器，AI 在里面随便折腾，但没有任何你的个人数据。安全是安全了，但能做的事非常有限——登录要重新来、Cookie 一个没有、付费墙全挡住。

然后是「云浏览器」方案——BrowserAct、Browserbase——在云端跑一个浏览器实例，AI 通过远程控制来操作。绕过了反爬、能做数据采集，但成本高、数据隐私风险大。你每一条 Cookie 都在云端流转，信任成本太高了。

2026 年，一个新的路线冒出来了：**借用真实浏览器**。Kimi WebBridge 和腾讯 BrowserSkill 是两个代表。核心思路是——不另起炉灶了，直接接入你正在用的那个浏览器。你平时怎么上网，AI 就怎么借你的环境去上网。你登录过的东西 AI 自动能访问，你有的权限 AI 自动有，你在哪个标签页 AI 就能接管哪个标签页。

这不只是一个技术优化，而是一个设计哲学的转向：从「给 AI 一个浏览器」到「AI 借用你的浏览器」。AI 不需要重建你的数字身份——你的浏览器就是你的数字身份，AI 只是暂时坐在你的位置上。

## 四、推荐：Kimi WebBridge

市面上做 WebBridge 的产品有几个，我主要用的是 Kimi 的。理由很简单——

**它不只给 Kimi 自己用。**

Kimi WebBridge 本质上是一个本地 HTTP 服务（跑在 `localhost:10086`），任何 Agent 工具只要会发 HTTP 请求，就能调用它。这个设计很聪明——Kimi 做了一个通用组件，但没把它锁在自己生态里。

实际效果是什么？你可以在 WorkBuddy 里调用它、在 Cursor 里调用它、在任何一个装了对应 Skill 的 Agent 里调用它。它不关心你是谁家的 Agent，它只管「收到指令 → 操作浏览器 → 返回结果」。

这也是我选它的核心原因——我不需要一个只能给 Kimi 本人用的浏览器工具，我需要一个能接到我整个工作流里的浏览器工具。

**怎么装？** Kimi 官网有完整的安装教程，一步步照着来就行。本质上就是装一个浏览器扩展 + 本地起一个守护进程，五分钟搞定。不用我去复述——官网教程写得比我清楚。

## 五、我是怎么把它接到 WorkBuddy 里的

我把 Kimi WebBridge 做成了一个 Skill，挂在了 WorkBuddy 上。写技能的时候做了两件事：

第一，把调用指令写进了 WorkBuddy 的记忆系统里。这样 Agent 不需要每次等我手动告诉它「你现在可以用浏览器了」——它自己就知道，当任务需要浏览器访问权限的时候，自动去调那个 Skill。

第二，给它加了一个特殊用法。WorkBuddy 经常帮我生成 HTML 页面——你看到的这些文章页面，都是 WorkBuddy 生成的。以前我只能靠 Agent 自己说「嗯，生成好了，应该没问题」。但现在不一样了——Agent 写完 HTML 之后，我让它调 WebBridge 打开那个本地文件，截一张图回来，自己看一眼渲染效果是不是对的。CSS 有没有崩、排版有没有乱、颜色对不对——它自己验证自己。

这比「Agent 说搞定了我就信了」靠谱太多。它从一个只会写代码的助手，变成了一个会验收成果的助手。这个能力不是多高级的技术，就是给 Agent 多配了一双眼睛。

## 六、选对眼睛比选对模型更重要

写到最后想到一句话，应该是这篇的核心：

**给 AI 一个好模型只是给了它一个聪明的大脑。给它 WebSearch 是给它一本电话簿，给它 WebFetch 是给它一台不能装 APP 的功能机，给它 WebBridge 才是给了它一部你正在用的智能手机。**

三种工具不是互相替代——是递进的。轻量的问题用 WebSearch（搜个常识几毫秒就回来），需要深度阅读的用 WebFetch（把整篇文章读进来），被登录态或 JS 挡住的用 WebBridge（带着你的权限硬闯）。一条理想的调研流程是三层接力：搜 → 读 → 闯。

至于 BrowserSkill 和 WebBridge 的区别——腾讯开源的那个工具我还没试。但它 MIT 开源这一点对生态很重要：意味着未来任何 Agent 框架都可能直接内置类似的浏览器借用能力。这个方向是确定的，剩下的只是谁的实现更稳、API 更好用。

> 你给 AI 配的眼镜，决定了它能看到多大的世界。
