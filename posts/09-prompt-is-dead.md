## 一、什么是提示词——以及当年为什么有那么多人教写它

2022 年 11 月 ChatGPT 上线，所有人都可以跟大模型对话了。但同样一个问题，表达方式不同，答案质量天差地别——写"帮我写个工作总结"，模型给你一段干巴巴的流水账；写清楚面向谁、聚焦什么、语气如何，出来的东西就是另一个档次。**你说得越清楚，模型干得越接近你想要的样子。**

这就是提示词的本质：**你跟大模型之间的翻译层。** 模型是个纯函数，没有你的业务背景、不知道你的偏好。你需要把所有这些都写成文字放进 prompt 里。

于是 2023 年出现了"提示词工程师"这个头衔——把"怎么跟模型说清楚要求"变成了一门手艺。网上开始流传各种"咒语"：角色扮演（"你是一个有十年经验的律师"）、思维链（"让我们一步一步思考"）、少样本示例、分隔符技巧。当时这些确实有效——GPT-3.5 指令跟随弱，没有这些盔甲模型很容易跑偏。所以 2023 年教提示词，是在教你怎么给笨一点的模型打字。

到了 2026 年，模型的底层能力提升了一个量级。你不再需要花一半 token 预算给模型搭"人设"，对话接口也从协议层隔离了用户指令和外部内容。提示词从"一门手艺"退化成了"一个基本操作"——不是死了，是被模型吃掉了。

下面展开说说：哪些死了，哪些没死，以及为什么。

- **死了的，是那些用话术补模型短处的技巧**——模型和产品层把那些短处消化掉了：角色前缀（"你是一个有用的助手"）、分隔符防注入、"让我们一步一步思考"、堆一堆 few-shot 示例、反复强调"请输出 JSON"——这些 2023 年的咒语，2026 年的前沿模型基本不需要了。

- **没死的，是那些适应 LLM 物理性质的技巧**——物理性质没变，技巧就不会过时：长度按句不按字数（tokenizer 数不准字）、新任务开新会话（注意力会衰减）、给图片加文字锚点（视觉模型按显眼程度分配注意力）、先让它反对自己再下结论（RLHF 让模型爱谄媚）。

判断标准一句话：**这条技巧是在补模型的 bug，还是在适应模型的物理性质？** 补 bug 的会被版本号吃掉，适应物理性质的不死。

## 二、三年时间线：从 Prompt 到 Harness

先建个坐标系。三年下来，行业实际上经历了三个阶段：

| 阶段 | 时期 | 核心问题 |
|------|------|---------|
| **Prompt Engineering** | 2022–2024 | "我说什么，模型才听懂？" |
| **Context Engineering** | 2024–2026 | "模型该看到什么？什么时候看到？" |
| **Harness Engineering** | 2025–现在 | "整个 Agent 闭环怎么设计？" |

2023 年你在学怎么写好一句话。2026 年你在设计一个 Agent 系统——里面嵌着检索、记忆、工具调用、自我评估、护栏——提示词只是系统里靠近输入端的一小块。

Reddit 上有个高赞帖说得很直白："提示词工程已死，长寿者乃 context engineering。"另一篇文章说得更损："2024 年觉得自己背了一堆魔咒就是 prompt engineer；2026 年回头看，那只是 for 循环——必要，但远不是高级技能的分界线。"

所以吴恩达两门课讲的"提示词工程"四个字，在 2026 年的实际定位是：**整个系统里靠近输入端的一小块**。它不是没用，是被吸收进了更大的东西。

## 三、LLM 的三条物理性质

所有提示词技巧，追到底层都是在适应 LLM 的三条物理性质。把这些看懂了，技巧自己就能推出来。

1. **无状态。** Transformer 模型本身是一个纯函数——给它一段 token，输出下一个 token 的概率。它没有记忆变量，每次推理都是独立的前向计算。这个物理事实没变，也不会变。

> 产品层给你加了"伪记忆"——ChatGPT 的 Saved Memories、Claude 的 Projects、Kimi 的记忆系统——每次自动把"关于你的摘要"注入到 prompt 头部。日常聊天你确实不用每次自报家门了。但要记住三件事：一，记忆每次都要占 token；二，它"记住"的可能是错的；三，做严格的 Agent 系统时必须显式管理记忆，不能依赖黑盒。

2. **概率分布，不是确定输出。** 同样的话说两遍，结果大概率不同。temperature 控制发散度。2026 年推理模型（模型先内部"思考"几十到几百步再回答）让一致性提高了很多，但它仍然是采样过程，不是函数返回。工程上怎么应对？需要稳定时降 temperature、多跑几次取多数；需要发散时升 temperature、多采几次选最优。

3. **上下文窗口大了，有效注意力没大。** 窗口从 128K 扩到了 1M–10M，但 2023 年那篇 Lost in the Middle 的结论没变：放在中间的信息会被忽略。2025–2026 年被包了个新名字叫 **Context Rot**——在长上下文 benchmark 上，即使宣称 1M 窗口的模型，32K 之后性能就明显衰减。因为注意力机制是物理规律：序列越长，每个 token 分到的注意力预算越少。

所以不是窗口大就拼命灌 PDF——要**选择性灌入**。关键指令放开头或结尾（三明治布局），长任务定期做摘要，新任务开新会话——这是免费的性能优化。

## 四、知识源路由：问之前先选对路

吴恩达课程里最值钱的一条原则，不是你该写什么提示词，而是问问题之前先判断——**这条答案该从哪来？**

- **参数化记忆**：常识、概念解释、历史事实、语法——模型权重里有的东西。问"什么是熵"直接问就行。
- **检索记忆（RAG）**：私有资料、最新事实、需要引用的场景——从外部文档库拿，再丢给模型组织。问"我们公司去年财报如何"不能靠参数，会编。
- **工具调用**：实时数据、精确计算、实际操作——让模型调搜索/计算器/代码执行。问"今天上海天气""31415×92653"必须走工具。

这张路由表比任何提示词模板都重要。2026 年这套逻辑被称为 knowledge routing，是所有 Agent 系统的地基。

## 五、提示词四要素：2026 精简版

2023 年一篇好的提示词可能需要七八个要素堆叠。2026 年，对前沿模型来说，四个就够了——把它当写作业要求，别当念咒语：

1. **角色/受众**：什么时候才用？只有需要模型偏离默认助手人格时才指定（比如"以学术评审角度""面向小学生"）。普通任务不用。
2. **任务**：动词开头，简洁——"总结""对比""改写""提取以下字段"。
3. **约束/素材**：必要的上下文、限制、参考资料。素材放分隔符里。
4. **输出契约**：长度用**句/段/行**而不用字数（模型数不准）。需要机器消费时用 structured outputs。

一个 2026 风格的样例：

```
我要给团队写一段 release note。
素材：以下 3 条 commit 摘要 [...]
风格：面向非技术同事，避免 jargon
输出：3 段，每段 2 句，第一段是亮点
```

没有角色扮演，没有"step by step"，没有 5 个示例——但素材、形态、长度（按句）、受众全明确了。这就是 2026 年的提示词：更像写作业要求，不像念咒语。

## 六、可以丢掉的六条老技巧

吴恩达课程里强调过的这些东西，2026 年对前沿模型已经基本没用：

| 2023 在学的东西 | 2026 为什么可以扔 |
|---------------|-----------------|
| "You are a helpful assistant" 之类角色前缀 | 模型默认就是助手，除非让它偏离去当别的人格 |
| 用 `###` `"""` 分隔符防注入 | Chat 接口下用户消息和指令已自动隔离 |
| "Let's think step by step" | 推理模型已经内部 think，加了反而拖慢 |
| 每次重新自我介绍、说项目背景 | Memory/Projects 已跨会话记住 |
| 5-shot/10-shot 塞一堆示例 | 1-2 个足够，多了占 token 稀释注意力 |
| 反复强调"请输出 JSON" | Structured outputs / tool schema 已从协议层保证 |

简单说：2023 年你在用话术补模型的短处。2026 年那些短处大部分被模型和产品层消化了。

## 七、仍然值钱的四条

上面丢了六条，下面四条不但没丢，反而更重要：

**长度按句/段/行，不按字数。** 模型不能精确数词（tokenizer 的物理性质）。说"3 句"比说"50 字"靠谱十倍。这条三年没变。

**新任务开新会话。** 因为 Context Rot，长对话会污染注意力。新任务开新会话是免费的午餐。Claude 的 Projects、ChatGPT 的 Custom GPTs 本质上都在帮你做这件事——把上下文按任务分桶。

**给图片加文字锚点。** 传一张图时说"图中左上角的折线"远比"图里那个东西"准。视觉模型按"显眼程度"分配注意力，不是按你的意图——必须靠文字提供指针。

**先让它反对自己，再让它给结论。** 模型有谄媚倾向（你说什么它都点头）。对抗方法：先要求"列出对此方案的三条最强反对意见"，再让它综合判断。这条因为 RLHF 训练偏向取悦用户，在 2026 年反而更需要。

## 八、思维诱发：CoT 的 2026 用法

2022 年的经典发现——让模型先写推理步骤再给答案——到 2026 年怎么用：

- **用推理模型时**（o3、Claude thinking、DeepSeek R1）：不用手动加，它内部已 think；
- **用普通模型做数学/逻辑/代码**："先列推理步骤再答"仍然稳定提升正确率；
- **简单任务强行 CoT**：拖慢、占 token、纯浪费。

判断标准很简单：任务复杂到你自己也要停下来想想，再开 CoT。

## 九、失败模式速查表

| 症状 | 概率 | 处置 |
|------|------|------|
| 自信地说错事实（幻觉） | 中 | 切检索；要求附引用 |
| 遗忘早先指令 | **高** | 三明治布局；末尾再说一遍 |
| 长对话越来越笨（context rot） | **高** | 分阶段摘要；新任务开新会 |
| 对什么都点头（谄媚） | 中 | 先要求列出反对意见 |
| 输出长度偏离 | 中 | 改用句/段/行 |
| 执行了输入中暗藏的指令 | 中 | 分隔符圈输入；权限最小化 |

高概率的那两条——遗忘和 context rot——都是长上下文带来的副作用。窗口变大是好事，但你得知道怎么管理它。

## 十、一句话收尾

2023 年学提示词工程是在学一门手艺。2026 年学它，是在学一个系统组件。

底层那三条物理性质——无状态、概率分布、注意力衰减——不会变。把这三条吃透，模型怎么变你都能跟上。剩下的技巧，跟着版本号迭代就行。
